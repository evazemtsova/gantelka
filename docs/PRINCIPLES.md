# Принципы — общие для всех

Правила, которые действуют независимо от слоя (фронт / бек / инфра). Если работаешь в коде — сначала здесь, потом в специфичных правилах.

## 1. Reuse > rewrite

Прежде чем добавить новое — найди готовое. Дубль кода почти всегда хуже, чем некрасивое расширение существующего.

«Готовое» бывает разное:
- Компонент UI / эндпойнт / модель → переиспользовать
- Утилита / хелпер → переиспользовать
- Похожая логика → вынести в общий хелпер

Если переиспользовать неудобно — поправь оригинал, а не плоди второй.

## 2. Single source of truth

Одна сущность живёт в одном месте.

- Типы данных — в одном модуле
- Константы и подписи — в одном модуле
- Конфигурация — в одном файле
- Бизнес-правила — в одном слое (бек, не фронт)

Локальная копия типа / константы — баг. Расхождение копий — вопрос времени.

## 3. Don't build for hypothetical

Нет фичи в PRD — нет кода под неё. Не делай «на всякий случай». Не пиши «гибко, чтобы потом расширить». Расширим, когда понадобится.

Применимо к: лишним параметрам в функциях, конфигам, абстрактным классам, generic-обёрткам, флагам фичей без задачи.

## 4. Не глушить типы

В TypeScript: никаких `any`, `as any`, `// @ts-ignore`. Если приходится — комментарий почему.

В Python: тайп-хинты обязательны для публичных функций. `# type: ignore` только с комментарием.

Цель: компилятор/линтер — союзник, а не помеха. Если он мешает — значит, есть реальная проблема.

## 5. Не оставлять мусор

- Закомментированный код → удалить. Git помнит.
- Неиспользуемые импорты / переменные / экспорты → удалить.
- TODO без даты и автора → не писать. Лучше создать задачу.
- Логи дебага → удалить перед коммитом.

## 6. Комментарии — только когда WHY непонятен

Хороший комментарий объясняет «почему именно так», а не «что делает код». Если комментарий повторяет имя функции — удали.

Плохо: `// добавить элемент в массив` над `arr.push(x)`.
Хорошо: `// ID берём из time, а не uuid — нужна сортируемость по созданию`.

## 7. Перед коммитом

1. Тайп-чек проходит (frontend: `tsc`, backend: `mypy`/`pyright`/что выбрано)
2. Линтер проходит
3. Билд проходит
4. Изменилось user-facing поведение → запись в [CHANGELOG.md](./CHANGELOG.md)
5. Изменилась архитектура / контракт → обновлён [TDR.md](./TDR.md)

## 8. Перед удалением фичи

1. Удалить код (TS/CSS/Python — всё)
2. Удалить упоминания из типов / API
3. Удалить тесты этой фичи
4. Удалить из PRD (или пометить «вырезано — почему»)
5. `grep -rn "feature_name"` → 0 результатов
6. Запись в CHANGELOG: что убрали и **почему**

Шестой пункт — самый важный. Без него через полгода кто-то вернёт фичу как «потерянную».

## 9. Когда дёргаешь основу — проверь всех

| Меняешь | Проверь |
|---|---|
| Domain-тип | Всех, кто этот тип использует |
| API-эндпойнт / сигнатуру функции | Все вызовы |
| Дизайн-токен (цвет / шрифт) | 3+ страницы визуально |
| Database schema | Миграция + все запросы |
| Reducer-action / public method | Все диспатчи / вызовы |

## 10. Безопасность по умолчанию

- Не коммить секреты (`.env`, ключи) — `.gitignore`
- Не логировать чувствительное (пароли, токены, личные данные)
- Валидировать всё, что приходит от пользователя или внешнего сервиса
- На бэке — не доверять фронту, проверять права на каждом запросе

## 11. Документировать решения, не процесс

Запись в CHANGELOG нужна тогда, когда через 3 месяца кто-то спросит «почему это сделали?» Если изменение тривиально (опечатка, форматирование) — запись не нужна.
